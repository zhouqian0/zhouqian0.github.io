angular 的变更检测

Angular 引入 Zone.js 以处理变更检测 (ngZone)。ngZone 使 angular 可以决定何时需要刷新UI。

当我们试着在 angular 项目中将 main.js 中的 ngZone 移除之后，angular 的页面就不再会自动的更新数据。

// main.js
import {enableProdMode} from '@angular/core';
import {platformBrowserDynamic} from '@angular/platform-browser-dynamic';

import {AppModule} from './app/app.module';
import {environment} from './environments/environment';

if (environment.production) {
  enableProdMode();
}

platformBrowserDynamic().bootstrapModule(AppModule,
  {ngZone: 'noop'})
  .catch(err => console.error(err));

但是当我在组件中引入了一个 ApplicationRef 对象之后，并且在需要发生页面数据变更的时候调用其的 tick() 方法，奇迹出现了，页面又能够和往常一样更新数据了。

import {ApplicationRef, Component} from '@angular/core';

@Component({
    selector: 'app-root',
    template: `
        <div>{{count}}</div>
        <button (click)="add()">add</button>
        <button (click)="reset()">reset</button>
    `
})
export class AppComponent {
    count = 0;

    constructor(private applicationRef: ApplicationRef) {
    }

    add() {
    this.count++;
    this.applicationRef.tick(); //调用后页面中现实的的 count 值 + 1 
    }

    reset() {
    this.count = 0;
    this.applicationRef.tick() // //调用后页面中现实的的 count 值归零
    }
}

这是为什么呢？原来 NgZone 只是一种便捷机制，它是通过自动调用 applicationRef.tick() 这个方法，来实现变更检测，刷新页面数据的。

在浏览器中打上断点，进入到 tick() 这个方法之后就会发现，它遍历了所有的 view，并且对它进行了数据的变更检测，如果发生了数据对象的更新，就回去更新页面。

  tick(): void {
    if (this._runningTick) {
      throw new Error('ApplicationRef.tick is called recursively');
    }

    const scope = ApplicationRef._tickScope();
    try {
      this._runningTick = true;
      this._views.forEach((view) => view.detectChanges());
      if (this._enforceNoNewChanges) {
        this._views.forEach((view) => view.checkNoChanges());
      }
    } catch (e) {
      // Attention: Don't rethrow as it could cancel subscriptions to Observables!
      this._zone.runOutsideAngular(() => this._exceptionHandler.handleError(e));
    } finally {
      this._runningTick = false;
      wtfLeave(scope);
    }
  }

在网上查阅了相关的资料并且自己亲身实践之后找到了 detectChanges() 这个方法最终用以判断对象是否变更的函数 looseIdentical，detectChange() 函数中执行的 checkAndUpdateView() 函数就是依靠 looseIdentical() 这个函数来判别就对新旧对象之间是否更新的。

export function looseIdentical(a: any, b: any): boolean {
  return a === b || typeof a === 'number' && typeof b === 'number' && isNaN(a) && isNaN(b);
}

export function checkBinding(
    view: ViewData, def: NodeDef, bindingIdx: number, value: any): boolean {
  const oldValues = view.oldValues;
  if ((view.state & ViewState.FirstCheck) ||
      !looseIdentical(oldValues[def.bindingIndex + bindingIdx], value)) {
    return true;
  }
  return false;
}

看完源码之后再来聊聊 angular 的变更检测的触发场景吧。angular 的 变更检测触发场景有三：

  • 所有的 DOM 事件：click, change, input, …
  • XMLHttpRequest：从服务器获取数据
  • Timers：setTimeout(), setInterval()

看到这些场景，再一思索就会发现，这些个场景不都是异步事件吗，在根据这个思路再结合网上查找的资料就找到这个：onMicrotaskEmpty，这个函数在 angular 的官方文档中也有描述，大概就是等待 event loop 中的 micro task 队列清空之后触发，在 ApplicationRef 中对其订阅 进而去更新页面数据。

/**
 * Notifies when there is no more microtasks enqueued in the current VM Turn.
 * This is a hint for Angular to do change detection, which may enqueue more microtasks.
 * For this reason this event can fire multiple times per VM Turn.
 */
readonly onMicrotaskEmpty: EventEmitter<any>;

// ApplicationRef
this._zone.onMicrotaskEmpty.subscribe(
        {next: () => { this._zone.run(() => { this.tick(); }); }});

那么问题来了？为什么是 microtaskEmpty 呢，我在网上查找了一些文章后发现了这个：HTML Standard - event loop processing model ，这个一个来描述浏览器的事件循环的进程模型，在这篇文档里大致解释了原因：在页面更新渲染之前的上一步就是执行 microtask（涉及相关的 micro task 和 macro task 本文不作介绍）。而且触发这个 onMicrotaskEmpty 的函数也叫 checkStable()。

/*
  Notifies when the last onMicrotaskEmpty has run and there are no more microtasks, 
  which implies we are about to relinquish VM turn. 
  This event gets called just once.
*/
function checkStable(zone: NgZonePrivate) {
  if (zone._nesting == 0 && !zone.hasPendingMicrotasks && !zone.isStable) {
    try {
      zone._nesting++;
      zone.onMicrotaskEmpty.emit(null);
    } finally {
      zone._nesting--;
      if (!zone.hasPendingMicrotasks) {
        try {
          zone.runOutsideAngular(() => zone.onStable.emit(null));
        } finally {
          zone.isStable = true;
        }
      }
    }
  }
}

讲完异步触发的场景，再来看 component 初始化加载时页面是为什么刷新的。看了一看 ApplicationRef 这个类，就找到了这个 _loadComponent，并且这个函数也调用了 tick 函数，且引用这个 _loadComponent 函数的函数叫 bootstrap ,这是个引导加载 component 的函数。

  private _loadComponent(componentRef: ComponentRef<any>): void {
    this.attachView(componentRef.hostView);
    this.tick();
    this.components.push(componentRef);
    // Get the listeners lazily to prevent DI cycles.
    const listeners =
        this._injector.get(APP_BOOTSTRAP_LISTENER, []).concat(this._bootstrapListeners);
    listeners.forEach((listener) => listener(componentRef));
  }

  private _unloadComponent(componentRef: ComponentRef<any>): void {
    this.detachView(componentRef.hostView);
    remove(this.components, componentRef);
  }

在接下来讲讲如何主动去触发页面的检测吧，讲到这个就要介绍下 ChangeDetectorRef。

export abstract class ChangeDetectorRef {
  abstract markForCheck(): void;
  abstract detach(): void;
  abstract detectChanges(): void;
  abstract reattach(): void;
}

这里介绍下这个几个方法吧:

  • markForCheck() - 将视图标记为已更改，并在下次 applicationRef.tick() 时进行更新，需设置组件的 ChangeDetectionStrategy 为 ChangeDetectionStrategy.OnPush
  • detach() - 从变化检测树中分离变化检测器，该组件的变化检测器将不再执行变化检测，除非手动调用 reattach() 方法。
  • reattach() - 重新添加已分离的变化检测器，使得该组件及其子组件都能执行变化检测
  • detectChanges() - 从该组件到各个子组件立刻执行一次变化检测

这里常用的方法是 markForCheck() 和 detectChanges()，着重介绍下这两个函数的差异吧，当页面组件调用 markForCheck() 时会讲当前组件标记，标记后需要等到下一次 applicationRef.tick() 执行时才会去更新页面，在这之前，无论调用多少次 markForCheck() 都没有关系，因为只会执行一次。但是 detectChanges() 不会，每当调用一次 detectChanges() 便会立即触发页面的检测更新，检测当前组件及其子组件。

如果想要更可视化的了解变更检测可以试试这个工具，他可以更形象的表述变成检测从 root 到 子组件的变化。

如果想要关闭 angular 自己的变更检测，可以试试以下方法

  1. 执行于 zone.js 之外
    • this._zone.runOutsideAngular(()=>{…});
  2. 关闭 changeDetection
    • this._changeDetectionRef.detach();
  3. 从 main.js 中将 ngZone 移除（src/polyfills.ts 中也需要同步移除对 ngZone 的导入）
    • platformBrowserDynamic().bootstrapModule(AppModule,{ngZone: ‘noop’})

脏值检测

ExpressionChangedAfterItHasBeenCheckedError，这个错误想必用过 angular 的人基本都遇到过，那么为什么会出现这个错误呢？这里需要讲下 angular 使用的数据流：单向数据流，它指的是从组件树的顶部到底部渲染扫描过程中应用程序数据流转到由渲染过程生成的输出 DOM 数据结构的流程。并且在开发环境下，这个数据流在一次变更检测中会出现两次。所以一旦我们在开发环境下，且对一个数据在同一次变更检测中修改了两次值，导致它两次数据流扫过这个数据值时值不统一，故而出现了这个错误ExpressionChangedAfterItHasBeenCheckedError,感兴趣的话可以去看看 stackblitz 的 demo。那么可能有用过 angularJs 的人会有疑问，为什么 angularJs 中没有这个错误呢，那是因为 angularJs 使用的不是单向数据流，它会再一次检测中进行十数次的数据检测，这也造成了 angularJs 的性能下降。

@Component({
  selector: 'app-root',
  template: `
      <app-test [title]="title"></app-test>
  `
})
export class AppComponent {
  title: string;

  constructor() {
    this.title = "test"
  }

}

@Component({
  selector: 'app-test',
  template: `
      <p>{{title}}</p>
  `
})
export class TestComponent implements OnInit, AfterViewInit, AfterContentInit, AfterContentChecked {

  @Input() title: string;

  ngOnInit(): void {
    // this.title = "test2" // 正常
  }

  ngAfterViewInit(): void {
    this.title = "test2" // 会出现 ExpressionChangedAfterItHasBeenCheckedError
  }

  ngAfterContentInit(): void {
    // this.title = "test2" // 正常
  }

  ngAfterContentChecked(): void {
    // this.title = "test2" // 正常
  }
}

为什么在 ngAfterViewInit 的时候修改了数据会出现 ExpressionChangedAfterItHasBeenCheckedError 呢？因为在 view init 的时候已经经历过一次数据流，但在此时又对其中的数据进行了修改，造成了第二次数据流再扫过时数据不一致。但在 OnInit，AfterContentInit, AfterContentChecked 中修改数据，其实都算是在一次的数据流中。

那如何去避免这个错误呢？简单列举两种方法：

  • 使用 setTimeout()，将这次的数据修改放到下次的变更检测中。
  • 使用 changeDetectorRef 的 markForCheck() / detectChanges() 来标记页面重新检测变更 / 立刻重新更新页面
ngAfterViewInit(): void {
    // setTimeout(() => {
    //   this.title = "test2"
    // })
    this.title = "test2";
    this.changeDetectorRef.detectChanges();
    // this.changeDetectorRef.markForCheck()
  }

如果想关闭开发环境中的两次数据流检测可以修改 main.js 文件中的内容。

// main.js
import { enableProdMode } from '@angular/core';
import { platformBrowserDynamic } from '@angular/platform-browser- dynamic';
import { AppModule } from './app/app.module';
enableProdMode(); //启用产品环境
platformBrowserDynamic().bootstrapModule(AppModule) .catch(err => console.error(err));

路由器预加载策略

路由器预加载策略：ngx-quicklink。它提供了路由器预加载策略，该策略可自动下载与屏幕上所有可见链接关联的延迟加载模块。这里的预加载涉及到了[异步路由概念](https://www.angular.cn/guide/router#milestone-6-asynchronous-routing。

既然提到了预加载就要讲讲懒加载，因为没有懒加载就没有预加载，所谓的预加载就是在浏览器空闲的时候去加载懒加载模块。那么来讲讲懒加载吧，它有什么优点呢？当我们在使用懒加载时，它加载必须的模块，这就会让首次加载的包体积将会大大减小，加快应用的启动速度。那相应的，缺点就是，当懒加载的包过大时，加载的时候会相对较慢，影响用户体验。正是因为有这个缺点，预加载这个概念就诞生了，他解决了懒加载缺点，让浏览器在空闲的时候先吧懒加载的包慢慢加载起来，进而提升用户点击加载这个模块时的速度和体验。

angular service worker

首屏性能优化（SSR 服务器端渲染）

什么是服务端渲染（SSR）？服务端渲染是当用户第一次请求页面时，由服务器把页面渲染成 html 字符串，然后把它返回给客户端。客户端可以直接将其边解析边渲染，更快的将内容呈现给用户，而不需要为了生成 dom 再去运行 js 逻辑。

使用 SSR 渲染页面可以让用户可以减少因首屏加载而等待的时间。 但是由于是由服务器渲染 view 成 html 字符串发送给客户，这便会导致服务器的压力增大，增加了服务的成本。其次，由于页面是通过服务器计算而来的，发送给客户端需要一定的时间，故而客户端接受到第一个 bit 的时间便会延长，增加了 First Content Paint 的时间。

图像优化

首先 mozilla 已经有一篇系统的文章阐述响应式图像了。先推荐两个可以在线优化图片的工具 svgomg 和 squoosh。

再来讲讲 WebP，这是一种由谷歌开发的一种图像格式，这种格式对比 JPEG ,PNG 体积能少 25%-35%，从而显著优化页面加载时间和带宽使用情况。具有更优的图像数据压缩算法，能带来更小的图片体积，而且拥有肉眼识别无差异的图像质量；同时具备了无损和有损的压缩模式、Alpha 透明以及动画的特性，在 JPEG 和 PNG 上的转化效果都非常优秀、稳定和统一。

WebP 的缺点就是解码速度相对于 JPEG ,PNG 慢，但是由于其带来的更小的体积缩短了加载时间，其实文件渲染速度然而是更快了。其次就是它目前支持了市面上的约 78% 的浏览器，还有部分浏览器尚未支持，如safari。

再者对图像而言，压缩色彩质量至原来的 80%-85% 将会将体积压缩 30%-40%，且对图像的影响最小。

还有就是在 img 标签上使用 loading="lazy” 可以将图片设置为懒加载，如 Medium 就是将图片设置为该格式，但是缺点就是该格式目前只支持 Chrome。

异步加载 js

在 script 标签中使用 defer 可以讲不需要立刻使用的 js 推迟加载。

<script defer src="https://www.googletagmanager.com/gtag/js?id=xxx">
</script>

字体

对字体使用 swap 可以为字体提供一个非常小的阻塞周期和无限的交换周期。可以设置让程序加载时使用默认字体，然后等原本指定的字体 ready 之后 将其 swap（因为原本的默认字体加载速度一定会比非默认字体加载快）。

@font-face {
  font-family: Source Sans Pro', sans-serif;
  src: url('...') format('woff');
  font-display: swap;
}
